Spark Mental Models
Phase 2 — Computational Thinking

1. Lazy eval — transformations vs actions

Transformations (lazy — return a DataFrame, nothing runs):

select filter withColumn join groupBy().agg() repartition

Actions (eager — trigger a job):

count collect show write toPandas take

One-liner
"Transformations just extend the plan — the cost only lands when I call an action. Catalyst sees the whole chain first so it can reorder, prune, and pick join strategies."
Trap
count() in a loop re-runs the entire upstream pipeline every time. Call it out if you see it.

2. Partitions — the unit of parallelism

1 partition = 1 task = 1 core. Aim for low hundreds of MB each.

  • Too few → idle cores, memory pressure
  • Too many tiny → scheduling overhead, small-files problem

spark.sql.shuffle.partitions = 200 (default). AQE coalesces down automatically.

3. Narrow vs wide

TypeMovementExamples
NarrowNo data moves (stays on machine)select filter withColumn union
WideShuffle — data crosses networkgroupBy join distinct orderBy
One-liner
"Narrow stays on the same machine. Wide shuffles across the network — and the shuffle is where time and money go."

4. Jobs → Stages → Tasks

  • Job — one action call
  • Stage — split at every shuffle boundary (Exchange node in plan)
  • Task — one partition, on one core, in one stage

"How many stages?" ≈ "How many shuffles?" Fewer = faster.

Diagnose: df.explain("formatted") — look for Exchange, SortMergeJoin, missing PushedFilters.

5. Catalyst & AQE

Catalyst — rewrites the logical plan before execution:

  • Predicate pushdown to scan
  • Column pruning
  • Join strategy selection

AQE (on by default) — re-optimizes during execution:

  • Coalesces shuffle partitions
  • Promotes sort-merge → broadcast at runtime
  • Splits skewed partitions
Talking point
"AQE handles partition coalescing and skew, but it can't fix bad logic — it won't avoid a shuffle I didn't need, or fix a Python UDF."

6. Tungsten / Photon

Tungsten — compact binary formats + whole-stage codegen (JVM). Photon — Databricks native C++ vectorized engine. Both break when you use a Python UDF (data must serialize out to Python row-by-row). Use built-ins to stay in the fast path.

7. Joins

StrategyWhenCost
Broadcast hashOne side small (≤~10 MB default; AQE can trigger)Cheap — only small side ships; big side stays put
Sort-mergeBoth sides largeShuffles & sorts both sides

Force it: fact.join(broadcast(dim), "id")

8. Driver vs executors

collect() / toPandas() pulls all rows to the driver. On large data → OOM. Use show(n) / limit(n).show() or write instead.

Narrate it back — "Walk me through what happens when I call df.groupBy('country').agg(sum('amount')).count()"
groupBy and agg are transformations → they extend the logical plan. count() is the action → Catalyst optimizes the whole chain (pushes filters, picks join strategy) → physical plan produced with Exchange nodes marking shuffle boundaries → job submitted → stages cut at each Exchange → each stage runs one task per partition in parallel across cores → count result returned to the driver.
Optimization Levers
Phase 2 — find 4–6 of these in the slow app
1Broadcast joins — eliminate the big-side shuffle
Symptom: plan shows SortMergeJoin + Exchange on both sides; one side is a small lookup table.
# BEFORE
result = fact.join(dim, "product_id")   # shuffles 100M rows

# AFTER
from pyspark.sql.functions import broadcast
result = fact.join(broadcast(dim), "product_id")
Watch out: only broadcast if the small side truly fits in executor memory (≤ ~100 MB). AQE may auto-broadcast, but an explicit hint is reliable.
"Instead of reshuffling 100M rows we send 5k rows to every machine — minutes vs seconds."
2Filter & project early — predicate pushdown
Symptom: select("*"), joins and aggregations, then a WHERE at the end.
# BEFORE
df = spark.table("events")              # 200 cols, all dates
out = df.join(other, "id").groupBy("country")
        .agg(F.sum("amount")).where(col("year")==2025)

# AFTER
df = (spark.table("events")
        .where(col("year")==2025)       # partition prune
        .select("id","country","amount")) # projection prune
"We were loading 200 columns for all time just to use 3 of them for one year."
3Fix partitioning — repartition vs coalesce
Symptom: 200 tiny tasks for a small dataset, or 4 giant tasks choking a stage.
# Reduce without shuffle (before write) — narrow, cheap
df.coalesce(8).write.format("delta").save(path)

# Redistribute WITH shuffle — use when increasing or de-skewing
df.repartition(200, "customer_id")

# Right-size for small dataset; AQE also does this
spark.conf.set("spark.sql.shuffle.partitions", 64)
"2,000 tiny output files make every downstream read slow — coalescing writes a handful of right-sized files."
4Replace Python UDFs with built-ins
Symptom: udf(lambda ...) doing what a built-in function could do.
# BEFORE — row-by-row serialization, blocks optimizer
upper_udf = udf(lambda s: s.upper(), StringType())
df = df.withColumn("u", upper_udf("name"))

# AFTER — vectorized, Photon-friendly
df = df.withColumn("u", F.upper("name"))

If custom logic is unavoidable → use a vectorized (pandas) UDF: batch Arrow, not row-by-row.

"That UDF hands every row to Python one at a time. The built-in processes whole batches in native code — same result, fraction of the time."
5Handle data skew
Symptom: one task runs 10× longer; one huge partition in spark_partition_id count.
# Diagnose (serverless-safe)
(df.groupBy(F.spark_partition_id().alias("pid"))
   .count().orderBy(F.desc("count")).show())

# Fix A: AQE (on by default — mention it)
# spark.sql.adaptive.skewJoin.enabled = true

# Fix B: broadcast the small side (avoids shuffle entirely)
big.join(F.broadcast(small), "key")

# Fix C: salt the hot key
salted = big.withColumn("salt", (F.rand() * 16).cast("int"))
"One customer has half the rows — one machine did half the work. Spreading the hot key balances load so all machines finish together."
6Don't recompute — materialize (no collect)
Symptom: same DataFrame used N times (re-runs lineage N×); or collect()/toPandas() on large data.
# SERVERLESS: materialize to Delta (not .cache())
base.write.mode("overwrite").saveAsTable("default.base_step")
base = spark.table("default.base_step")

# Never: rows = base.collect()   ← driver OOM
# Instead: base.limit(20).show()
.cache() is restricted on Databricks serverless/Free Edition — materialize to Delta instead. Saying this out loud is the scoring point.
"We were rebuilding the same intermediate result three times. One write = later steps near-instant, driver safe."
Delta Lake
Phase 2 — Code Stewardship

Why Delta exists

Plain Parquet = cheap columnar reads but no ACID, no updates, no time travel, no concurrent writers. Delta wraps Parquet with a transaction log (_delta_log/) that delivers all four.

MERGE — upsert pattern

MERGE INTO target t
USING source s ON t.id = s.id
WHEN MATCHED     THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;
  • Idempotent re-runs — if today's batch arrives twice, MERGE doesn't double-insert.
  • Predicate on the join key — without it, MERGE scans all target files.
  • Cost = rewriting matched files (copy-on-write). Many targeted updates → file churn → use Deletion Vectors.
"We track every change, so reports can answer 'what did the table look like on March 5?'"

Time travel

SELECT * FROM events VERSION AS OF 42;
SELECT * FROM events TIMESTAMP AS OF '2026-06-01';
spark.read.format("delta").option("versionAsOf", 42).table("events")
"If a bad batch corrupts the table we can roll the read back while we investigate — no restore-from-backup window."
Retention
Default 7-day file retention, 30-day log retention. Never VACUUM 0 HOURS — breaks active readers.

Schema evolution

# Add new columns on write
df.write.format("delta").mode("append")
        .option("mergeSchema", "true").saveAsTable("target")

spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true")

mergeSchema=true accepts new columns only, not type-incompatible changes. A renamed column = "drop + add" — usually wrong.

OPTIMIZE & VACUUM

OPTIMIZE my_table;
OPTIMIZE my_table WHERE date >= '2026-06-01';
OPTIMIZE my_table ZORDER BY (customer_id);

Rewrites many small files → fewer ~128 MB files. Run nightly on heavy-write tables. Storage doesn't shrink until VACUUM.

"Every future query gets faster — not just today's — because reads hit a handful of right-sized files instead of thousands of tiny ones."

ZORDER vs Liquid Clustering

ZORDERLiquid Clustering
SetupAfter every OPTIMIZEDeclared at table creation (CLUSTER BY)
MaintenanceRe-run manuallyAuto on OPTIMIZE, incremental
Changing keysFull reorder from scratchALTER TABLE … CLUSTER BY is cheap
Use whenOlder tables, classicNew tables on modern runtimes

Both enable data skipping: filters skip files whose stats don't overlap the predicate.

"We physically co-locate rows that get filtered together so the engine skips most files for a typical query."

SCD2 schema

business_key    -- natural id
<attrs…>        -- slowly-changing columns
effective_from  DATE NOT NULL
effective_to    DATE       -- NULL = current
is_current      BOOLEAN    -- denormalized convenience flag

SCD2 needs two MERGE passes — one to close old row, one to insert new row.

Things candidates get wrong

MistakeReality
"MERGE is faster than append"MERGE does more work (read + match + rewrite). It's correct, not faster.
"Partition by customer_id"High-cardinality partition → thousands of tiny folders. Use ZORDER/Liquid instead.
"ZORDER every column"Stats degrade past ~3 ZORDER columns. Pick what you actually filter on.
"Delta only works on Databricks"Open-source delta-spark runs on any Spark.
Narrate it back — "How would you handle late-arriving updates to a customer dimension table?"
I'd use a Delta SCD2 pattern — MERGE on the natural business key. Matched rows where something changed: close the current row by setting effective_to to today and is_current to false, then insert the new state. Unmatched rows insert directly. This is idempotent — re-running the batch won't duplicate rows because the MERGE condition guards it. I'd also make sure the MERGE predicate includes a date range filter so Delta's file pruning only touches recent files, not the whole table.
Python Feature Dev & Data Patterns
Phase 1 — Code Stewardship

5-step protocol for reading unfamiliar code

Say each step out loud — this is the performance, not just the prep.

  1. Find the entry point. What does a caller actually invoke? (run(), main(), the public method.)
  2. Trace the data contract. What goes in, what comes out, what shape? Name the types.
  3. Name the extension points. Where is this designed to be extended? (base classes, registries, callbacks.)
  4. Spot the invariants. What must stay true? (e.g., "every rule returns a result; results never raise.")
  5. Only then change it. Add the feature the way the code wants to be extended, not by bolting on a special case.
Scoring move
Saying "let me restate what this does before I touch it" is itself a point.

Medallion architecture

BronzeRaw, as-ingestedAppend-only; one source = one table
SilverCleaned, validated, deduped, joinedMERGE upserts; DQ checks

"I'd land raw into Bronze so we can replay if Silver logic changes; Silver is where dedup/SCD happens; Gold is what the dashboards read."

Idempotent incremental loads

Three ingredients for a safe re-runnable nightly job:

  1. Watermark column in source (updated_at)
  2. MERGE on natural key — not append (re-running won't duplicate)
  3. High-water-mark table — record max watermark processed per source
"The job is safe to re-run from scratch — worst case we redo work, never duplicate or skip rows."

Deduplication patterns

dropDuplicates(["id"]) keeps an arbitrary row when keys collide. Use the window pattern if you need "latest wins."

CDC ingestion shapes

Ask the customer: "Is the source a snapshot or an event stream? Is there a soft-delete flag?"

  • Snapshot per period — full dump every N hours → MERGE on natural key
  • Insert/update/delete events — op flag column → MERGE handling all 3 op types
  • Log-based CDC (Debezium/Fivetran) — before/after images → after-image + op flag is sufficient

6-point audit checklist

  • Does it run? Test on a small slice first.
  • Is it correct? Spot-check counts/rows against expected.
  • Is it efficient? Needless shuffle? Python UDF? collect()? distinct()?
  • Is it serverless-legal? No sparkContext / RDDs / .cache().
  • Are the APIs real? Function names, params, config keys — do they exist?
  • Did it preserve the contract? In Phase 1: same inputs/outputs, extension points kept?

All narrate-it-back prompts

Close the page. Answer in <45 seconds out loud. Check the answer.

1. What happens when I call df.groupBy('country').agg(sum('amount')).count()?
transformations build plan → count() triggers Catalyst optimization → physical plan with Exchange nodes → job → stages at shuffle boundaries → tasks per partition → result to driver
2. Why is filter cheap but groupBy expensive?
filter is narrow — stays on the same machine. groupBy is wide — Spark must redistribute all rows by key across the network (the shuffle). Shuffle writes to disk, crosses the network, re-reads — that's where time and money go.
3. Your job has 200 tasks but the data is tiny. What's happening and what fixes it?
spark.sql.shuffle.partitions defaults to 200 so the shuffle stage created 200 partitions even though the data is tiny — thousands of near-empty tasks with scheduling overhead dominating. Fix: lower the setting or let AQE auto-coalesce (it does this by default in modern Databricks).
4. When would you force a broadcast join, and what's the risk if the "small" side isn't small?
Broadcast when one side is a small lookup/dimension table (≤~100 MB) — the big side never shuffles. Risk: broadcasting something too large OOMs executors because every executor gets a full copy. Always sanity-check the actual size before hinting.
5. Why does a Python UDF make an otherwise-fast query slow?
A Python UDF serializes every row out of the JVM to a Python process and back — row-by-row. It breaks out of Catalyst optimization, whole-stage codegen, and Photon. It's often 10–100× slower than the equivalent built-in. If I can't avoid custom logic, I'd use a vectorized (pandas) UDF which works in Arrow batches instead.
6. One task in a stage runs 10× longer than the rest. What do you suspect and how do you fix it?
Data skew — one join/group key has vastly more rows than others. Check with spark_partition_id groupBy count. Fixes in order: let AQE skew-join handle it (it's on by default); broadcast the small side to avoid the shuffle entirely; salt the hot key to spread it across partitions.
7. Why is df.toPandas() dangerous on a large DataFrame?
It pulls every row from all executors into the driver's memory. On large data the driver OOMs or becomes unresponsive. The alternatives: write() for distributed output, or show(n)/limit(n).show() for a quick peek.
8. AQE is on by default. Does that mean you don't need to think about optimizations?
No. AQE handles partition coalescing, runtime broadcast promotion, and skew splitting — but it can't fix bad logic. It won't avoid a shuffle you introduced unnecessarily, fix a Python UDF blocking vectorization, or push a filter you placed after an expensive join. AQE is a safety net, not a replacement for knowing the levers.
9. Genie Code gives you code with df.cache() on Databricks Free Edition. What do you do?
Call it out — cache() is restricted on serverless/Free Edition. The fix is to materialize to Delta instead: write the DataFrame to a table, then read it back. That's the serverless-safe equivalent of caching and it actually gives you durable storage. I'd tell the interviewer: "Genie's answer would fail here — let me swap that out."
10. How would you make a nightly ingestion job safe to re-run?
Three ingredients: a watermark column in the source to identify new/changed rows, MERGE on the natural key (not append — so re-running doesn't duplicate), and a high-water-mark table recording the max watermark processed. On failure you reset the HWM and re-run — worst case you redo work, never duplicate or skip.
