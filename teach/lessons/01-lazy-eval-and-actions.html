# Lesson 01 setup — creates Delta tables used in the examples
# Run once per cluster session; safe to re-run (uses overwrite mode)

from pyspark.sql import functions as F
from pyspark.sql.types import *
from pyspark.sql.functions import broadcast
import random, datetime

spark.sql("CREATE DATABASE IF NOT EXISTS lesson_db")

# ── orders: 500k rows, partitioned by year, customer 1 is skewed ─────────────
orders_schema = StructType([
    StructField("order_id",    LongType(),   False),
    StructField("customer_id", IntegerType(),False),
    StructField("product_id",  IntegerType(),False),
    StructField("country",     StringType(), False),
    StructField("amount",      DoubleType(), False),
    StructField("year",        IntegerType(),False),
    StructField("order_date",  DateType(),   False),
])
rows = []
countries = ["US","US","US","US","GB","DE","FR","BR","IN","JP"]
for i in range(500_000):
    cid = random.randint(1, 200)
    if cid <= 5: cid = 1    # customer 1 = hot key for skew demos
    rows.append((
        i, cid, random.randint(1,100),
        countries[random.randint(0,9)],
        round(random.uniform(5.0, 2000.0), 2),
        random.choice([2023,2024,2025]),
        datetime.date(random.choice([2023,2024,2025]),
                      random.randint(1,12), random.randint(1,28))
    ))
orders_df = spark.createDataFrame(rows, orders_schema)
(orders_df.write.mode("overwrite").format("delta")
    .partitionBy("year")
    .saveAsTable("lesson_db.orders"))

# ── customers: 200 rows (small — broadcast candidate) ────────────────────────
customers_df = spark.createDataFrame(
    [(i, f"Customer {i}", ["US","GB","DE","FR","BR"][i%5])
     for i in range(1, 201)],
    ["customer_id","name","region"]
)
(customers_df.write.mode("overwrite").format("delta")
    .saveAsTable("lesson_db.customers"))

print("Setup complete.")
spark.sql("SELECT year, COUNT(*) as cnt FROM lesson_db.orders GROUP BY year ORDER BY year").show()

Why this lesson exists

A core interview signal is being able to explain how your code scales. Everything in Spark starts with one question: when does my code actually execute? If you can't answer that, you can't explain performance, you can't find bugs, and you can't impress an FDE who lives in this runtime every day.

This concept also directly explains the single most common interview trap: adding a count() "just to check" and accidentally triggering a full pipeline re-run.

The core model: Plan → Execute

Spark is lazy. When you write a transformation, Spark does not run anything — it records your intent as a logical plan. Execution only happens when you call an action.

One-liner for the interview "Transformations are lazy — they just extend the logical plan. The cost only lands when I call an action like write or count. This lets the optimizer see the whole chain before spending a single CPU cycle."

Transformations (lazy — return a DataFrame)

These build the plan. Nothing runs. Nothing is evaluated. You can chain hundreds and Spark will not move a byte:

# All of this is just a plan. Zero execution.
df = spark.table("lesson_db.orders")
filtered = df.filter(df.year == 2024)
selected = filtered.select("customer_id", "country", "amount")
customers = spark.table("lesson_db.customers")
joined   = selected.join(broadcast(customers), "customer_id")
grouped  = joined.groupBy("country").agg(F.sum("amount"))

Actions (eager — trigger a job)

These tell Spark: "Now execute the whole plan and give me a result." The moment you call one, a Spark job is submitted:

grouped.count()          # → triggers execution (returns a number)
grouped.show(10)         # → triggers execution (prints rows)
grouped.collect()        # → triggers execution (pulls all rows to driver)
grouped.write.format("delta").save("...")  # → triggers execution
grouped.toPandas()       # → triggers execution (dangerous on large data!)
⚠ Common interview trap count() inside a loop, or as a "quick sanity check," executes the entire upstream pipeline every time. It's a silent performance killer. If you spot this in the slow app you're given — call it out.

Why laziness is actually powerful

Lazy evaluation gives the Catalyst optimizer a complete picture of your intent before it runs anything. With that global view it can:

  • Push filters down — apply WHERE year = 2024 at the file scan, reading only the rows you need
  • Prune columns — skip columns that are never referenced downstream
  • Reorder operations — move cheap operations (filters) before expensive ones (joins)
  • Choose join strategies — decide broadcast vs sort-merge based on table sizes

This is why declarative DataFrame/SQL code beats hand-rolled Python loops: the optimizer can only help you if you express intent as a plan, not as imperative steps.

Interview talking point "Catalyst can't optimize what it can't see. If I drop to a Python loop or a regular UDF, I break out of the optimized path. That's one reason I always prefer built-in Spark functions — they stay inside the plan."

Practical: reading your own plan

You can inspect the plan Spark built without executing anything:

# Build the plan and inspect it — zero execution until an action
df = spark.table("lesson_db.orders")
plan_df = df.filter(df.year == 2024).select("customer_id", "country", "amount")

# Shows Parsed → Analyzed → Optimized → Physical plan
plan_df.explain("formatted")

# Quick: just the physical plan (look for PushedFilters, Exchange nodes)
plan_df.explain()

In the physical plan, look for PushedFilters on scans (good — Catalyst pushed it down), Exchange nodes (shuffles — expensive), and BroadcastHashJoin vs SortMergeJoin.

Knowledge checks

Q1 — Which of the following triggers Spark to execute the pipeline?

  • df.filter(col("year") == 2024)
  • df.withColumn("full_name", concat("first","last"))
  • df.count()
  • df.join(other, "id")

Q2 — A teammate says "I added df.count() at the end of every stage so I can check row counts in the logs." What's the problem?

  • count() is deprecated — use len(df) instead
  • Each count() call triggers a full execution of the upstream plan, so N stages means N full re-runs
  • count() only works on cached DataFrames
  • count() is a transformation, not an action, so it accumulates memory

Q3 — Why does the Catalyst optimizer prefer you to use built-in Spark functions over Python UDFs?

  • Built-in functions are written in Python and therefore faster to parse
  • Python UDFs require a GPU to execute
  • Built-in functions stay inside the Catalyst/Tungsten optimized path; Python UDFs break out of it — Spark can't see into them, can't optimize them, and must serialize data row-by-row to the Python process
  • Python UDFs trigger an action on every row
